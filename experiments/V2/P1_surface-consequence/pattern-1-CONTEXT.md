# Pattern 1: Surface Fix, Missed Consequence

## What this pattern tests

Gemini Flash reliably solves problems that have a name (data leakage, Simpson's paradox, ANOVA violations). But it reliably misses the second-order step that follows the named fix, when that step is not itself a named technique.

This pattern is PROVEN. In v1, the data-leakage-detection task failed Gemini 0/3 with identical numeric output every trial. The model fixed all three leakage bugs correctly but missed that one column became useless noise after the fix. A senior data scientist would audit the post-fix feature set and drop it. Gemini doesn't.

## Design recipe (mandatory for every task in this folder)

1. Front-load with a NAMED problem the model will solve easily
2. Engineer an UNNAMED side effect of the correct fix
3. Make the verifier check only the side effect, not the named fix

The unnamed side effect must be:
- A natural result of the correct fix (not a separate planted bug)
- Visible only if the agent examines its work AFTER applying the fix
- Resolvable by a routine action a senior practitioner would do by habit
- NOT described anywhere as a standard step (no blog post says "after fixing X, always do Y")

## Harbor task structure (use for every task)

```
task-name/
├── instruction.md          # 2-3 paragraphs, clear, no hints about the side effect
├── task.toml               # schema_version = "1.1"
├── environment/
│   ├── Dockerfile          # python:3.11-slim + pandas numpy scipy scikit-learn statsmodels
│   └── (data files)        # generated by generate_data.py
├── tests/
│   ├── test.sh             # calls verify.py, writes /logs/verifier/reward.txt
│   └── verify.py           # checks the CONSEQUENCE, not the procedure
└── solution/
    └── solve.sh            # reference solution that does the named fix AND the audit step
```

Key rules:
- test.sh writes 1 or 0 to /logs/verifier/reward.txt
- /tests is mounted only when verifier runs, agent cannot see it
- Never put ground truth in the Dockerfile
- Verifier checks substance (numeric output, structural properties), not procedure (code patterns, keywords)

## Build 6 tasks, described below

### Task 1: mnar-imputation-destroys-signal

**Named problem:** Dataset has 12% missing values in one column. Agent is told to handle missing data and run a regression to identify significant predictors.

**Unnamed side effect:** The missingness is MNAR (missing not at random). Values are missing specifically when they're high. Mean imputation collapses the real relationship between that column and the outcome, making a genuinely significant predictor appear non-significant.

**Data generation:** 2000 rows, 8 features, 1 target. Feature X3 has true coefficient ~0.45. Delete X3 values where X3 > 75th percentile with 60% probability. Mean imputation compresses X3's variance and kills the coefficient. Correct approach is to recognize MNAR (plot missingness vs other features or outcome), then either use a missingness indicator, multiple imputation with MNAR model, or pattern-mixture model.

**Verifier:** Run agent's script. Check that X3's regression coefficient is statistically significant (p < 0.05) and within [0.30, 0.60]. Mean imputation produces p > 0.15 and fails. MNAR-aware approach produces p < 0.01 and passes.

**Solution:** Detect MNAR via missingness correlation with outcome. Add a binary missingness indicator column. Impute with a value that doesn't compress variance (or use multiple imputation). Run regression with both the imputed column and the indicator.

### Task 2: onehot-rare-categories-overfit

**Named problem:** Buggy script one-hot-encodes a high-cardinality categorical BEFORE train/test split, causing leakage (test categories leak into the encoding). Agent must fix to encode after split.

**Unnamed side effect:** After the correct fix (encode after split), ~35% of the one-hot columns in training have fewer than 3 non-zero entries (rare categories). These columns are effectively noise and cause overfitting. The model trains on a sparse, near-empty matrix for those features and overfits.

**Data generation:** 5000 rows, 10 features + 1 high-cardinality categorical (city_name, 120 unique values, Zipf-distributed so 40 cities have < 5 rows each). Binary target with real signal from the numeric features only. The categorical has no real predictive power.

**Verifier:** Run agent's fixed pipeline on held-out test set. Check accuracy is in [0.72, 0.84]. A pipeline that fixes the split but keeps all one-hot columns overfits and gets ~0.65. A pipeline that also filters rare categories (min_frequency threshold) or uses target encoding gets ~0.78.

**Solution:** Fix the split order. Then filter the one-hot encoding to categories with >= 10 training occurrences, or switch to target encoding for the high-cardinality column.

---

### Task 3: outlier-removal-kills-minority-class

**Named problem:** Dataset has obvious outliers (values at 100x the normal range). Agent is told to clean outliers and build a binary classifier.

**Unnamed side effect:** The outliers are NOT random. They are disproportionately from the minority class (class 1, which is 8% of data). Removing outliers by a simple IQR or z-score rule drops 40% of class-1 samples but only 2% of class-0 samples. The classifier trained on cleaned data has good overall accuracy but terrible recall on class 1.

**Data generation:** 3000 rows, class imbalance 92/8. The minority class has a subpopulation with legitimately extreme feature values (they're real, not errors). Plant 15 additional genuine outliers in class 0 (actual errors) to make "remove outliers" seem like the right blanket move.

**Verifier:** Check classifier's recall on class 1 in a held-out test set. Must be >= 0.55. Naive outlier removal drops recall to ~0.30. Correct approach (remove only class-0 outliers that are actual errors, or use robust methods that don't disproportionately affect minority class) maintains recall >= 0.60.

**Solution:** Examine outliers by class before removing. Note that class-1 "outliers" are legitimate extreme values. Remove only the 15 genuine class-0 errors. Use SMOTE or class-weighted classifier to handle remaining imbalance.

### Task 4: normalization-destroys-temporal-feature

**Named problem:** ML pipeline has a feature scaling bug. Features are scaled using the full dataset stats (mean/std computed on train+test). Agent must fix to scale using training data only.

**Unnamed side effect:** One feature is "days_since_first_purchase" (a monotonically increasing temporal feature). Standardizing it correctly (train-only stats) is fine for training, but when applied to test data (which is chronologically later), the scaled values for test data are all > 2.0 (because the test data's raw values are naturally larger than training). The model has never seen scaled values that high during training, so this feature becomes extrapolation noise.

**Data generation:** 4000 rows sorted by date. 8 features + days_since_first_purchase. Train is first 3000, test is last 1000. The temporal feature's test values are 1.5-3.0 standard deviations above the training mean after correct scaling.

**Verifier:** Run agent's pipeline. Check test accuracy is in [0.74, 0.86]. A pipeline that fixes scaling but keeps the temporal feature as-is gets ~0.70 (out-of-distribution scaled values hurt). A pipeline that also bins the temporal feature, uses quantile transform, or clips scaled values gets ~0.80.

**Solution:** Fix the scaling to train-only. Then notice the temporal feature's test distribution is out of range. Apply quantile-based binning or winsorize the scaled values to handle the temporal shift.

### Task 5: multicollinearity-after-log-transform

**Named problem:** Agent is given a regression dataset where the residual plot shows clear heteroscedasticity. Standard fix is to log-transform the target variable. Instructions say to fix the heteroscedasticity and report model coefficients.

**Unnamed side effect:** Two features (X1 and X2) have a correlation of 0.45 in raw space. After log-transforming the target (the correct fix), the effective correlation between X1 and X2 in predicting log(Y) jumps to 0.92 because both features are approximately log-linear with Y. This creates severe multicollinearity. VIF for both features exceeds 15. The coefficient estimates become unstable and their standard errors inflate, making one appear non-significant when it's actually important.

**Data generation:** 1500 rows, 6 features. Y = exp(0.3*X1 + 0.25*X2 + 0.4*X3 + noise). X1 and X2 are moderately correlated (r=0.45). After log(Y) transform, the model becomes linear in X1, X2, X3 but the X1-X2 multicollinearity in log-space is severe.

**Verifier:** Check that all three of X1, X2, X3 are reported as significant predictors (p < 0.05 for each). Without addressing multicollinearity, X2 typically shows p > 0.10. With VIF check + PCA/ridge/dropping one, all three show significance. Also check that heteroscedasticity is addressed (residual Breusch-Pagan p > 0.05).

**Solution:** Log-transform Y to fix heteroscedasticity. Check VIF. Notice X1-X2 multicollinearity. Use ridge regression, PCA on correlated pair, or drop one and note the alias. Report correct coefficients.

### Task 6: deduplication-loses-valid-longitudinal-data

**Named problem:** Dataset has apparent duplicate rows. Multiple rows share the same patient_id. Agent is told to clean the data and build a model predicting readmission risk.

**Unnamed side effect:** The "duplicates" are not duplicates. They are repeated measurements over time (same patient, different visits). Deduplicating on patient_id (keeping first or last) destroys the longitudinal signal. The visit_number and days_between_visits features, which are the strongest predictors, become meaningless if you keep only one row per patient.

**Data generation:** 800 patients, 1-5 visits each, 2200 total rows. Columns include patient_id, visit_number, days_between_visits, lab_value_1, lab_value_2, readmitted (target). The strongest predictor of readmission is the CHANGE in lab values between visits and short days_between_visits. Single-visit patients have near-baseline readmission rates.

**Verifier:** Check model AUC on held-out test set. Must be >= 0.72. Deduplicating drops AUC to ~0.58 (random-adjacent). Keeping longitudinal structure and engineering delta features gets AUC >= 0.75.

**Solution:** Recognize that patient_id repeats are multiple visits, not duplicates (visit_number column is the clue). Engineer features like delta_lab_value = current - previous visit, visit_frequency. Train on per-visit rows with patient-level train/test split to avoid leakage.

## After building all 6 tasks

For each task:
1. Write generate_data.py to create the synthetic data with fixed random seed
2. Run generate_data.py and copy outputs into environment/
3. Run `harbor run -p <task> -a oracle` (must get reward 1)
4. Run `harbor run -p <task> -a nop` (must get reward 0)
5. CRITICAL: Also run a "named-fix-only" stub that applies the obvious fix but skips the audit step. This stub should get reward 0. If it gets reward 1, the verifier isn't tight enough.
